# 贡献指南与规范

欢迎为 faleUI 项目做贡献！为了保持代码质量和项目一致性，请仔细阅读以下规范。

## 📁 文档存放位置说明

- **外部贡献文档**（如本文件）：放在源代码根目录
- **开发文档**：请写入本库使用指南文档于 `docs/` 内，内部开发文档于 `devdocs/` 内。
  - API文档
  - 开发指南
  - 技术设计文档
  - 内部实现细节等

## 🔧 Git 提交流范

### 提交消息格式

```
<类型>(<作用域>): <简短描述>

[可选的详细描述]

[可选的关联信息]
```

### 类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(gui): 添加按钮组件` |
| `fix` | 修复bug | `fix(core): 修复渲染异常` |
| `docs` | 文档更新 | `docs(readme): 更新安装说明` |
| `style` | 代码格式调整（不影响功能） | `style(format): 统一代码缩进` |
| `refactor` | 重构代码 | `refactor(ui): 优化组件架构` |
| `perf` | 性能优化 | `perf(render): 提升渲染性能` |
| `test` | 测试相关 | `test(button): 添加单元测试` |
| `chore` | 构建过程或辅助工具变动 | `chore(gradle): 更新依赖版本` |
| `ci` | CI配置相关 | `ci(workflow): 修改构建流程` |

### 作用域说明

常用的作用域包括：
- `core` - 核心功能
- `gui` - GUI组件
- `render` - 渲染系统
- `event` - 事件系统
- `util` - 工具类
- `api` - API接口
- `config` - 配置系统
- `i18n` - 国际化
- `compat` - 兼容性

### 提交示例

**良好示例：**
```
feat(gui): 添加可拖拽窗口组件

- 实现基础窗口拖拽功能
- 支持边界检测和限制
- 添加拖拽时的视觉反馈效果

解决 #123
```

**不良示例：**
```
update code
fix bug
添加新功能
```

## 🔄 Pull Request 规范

### PR标题格式

```
<类型>: <简短描述> (#PR编号)

例如: feat: 添加配置菜单界面 (#45)
```

### PR描述模板

创建PR时请使用以下模板：

```markdown
## 变更描述
<!-- 请详细描述本次变更的内容和目的 -->

## 相关Issue
<!-- 如果有关联的issue，请填写 -->
- Closes #issue_number
- Related to #issue_number

## 变更类型
<!-- 请勾选适用的变更类型 -->
- [ ] 新功能 (feat)
- [ ] Bug修复 (fix)
- [ ] 文档更新 (docs)
- [ ] 代码重构 (refactor)
- [ ] 性能优化 (perf)
- [ ] 测试 (test)
- [ ] 其他 (请说明)

## 检查清单
<!-- 请确认以下事项 -->
- [ ] 代码符合项目编码规范
- [ ] 添加了必要的测试
- [ ] 更新了相关文档
- [ ] 通过了所有测试
- [ ] 没有引入新的警告或错误

## 测试说明
<!-- 请描述如何测试这些变更 -->

## 截图（如果适用）
<!-- 如果是UI相关的变更，请提供截图 -->
```

## 🏷️ 版本标签规范

### 版本号格式

采用语义化版本控制：`主版本号.次版本号.修订号[-预发布标识][+构建元数据]`

示例项目版本格式：`0.1d+26.1s2`
- `0.1d` - 主要版本标识
- `26.1s2` - 对应的Minecraft快照版本

### 标签命名规则

```
v<版本号>

例如：
v1.0.0+26.1s2
```

### 发布标签时机

1. **正式发布**：完成功能开发和测试后
2. **预发布**：需要测试但尚未稳定的版本
3. **里程碑**：重要功能完成节点

## 🛠️ 开发环境设置

### 必需工具

- Java 25 JDK
- IntelliJ IDEA (推荐)
- Git

### 项目配置

1. 克隆仓库
```bash
git clone https://github.com/deplayeris/faleui.git
cd faleui
```

2. 导入IDE
   - 使用IntelliJ IDEA打开项目
   - 等待Gradle同步完成

3. 编译项目
```bash
# Windows
.\gradlew build

# Linux/macOS
./gradlew build
```

## ✅ 代码质量要求

### 编码规范

- 遵循Java 25语言特性
- 使用有意义的变量和方法名
- 为所有public方法添加JavaDoc注释
- 保持一致的代码风格

### 测试要求

- 新功能必须包含单元测试
- 修复bug需要添加回归测试
- 确保所有测试通过后再提交

### 文档要求

- 公共API必须有完整的文档
- 重大变更需要更新README
- 复杂功能需要在docs目录添加详细说明
- 对于对该模组的内部开发的一些文档，应写入devdocs目录
